# 改进用户故事 (Improve User Story)

## Who
需要在讨论新想法或变更时，保持现有 BDD 风格用户故事处于最新状态的开发人员、产品负责人、QA 工程师或智能体。

## What
主动检测对话中何时引入了新的改进、边缘情况或功能变更，并更新相关的现有用户故事以反映这些变化。如有必要，它会修改 **As a / I want / So that** 核心语句，并仔细添加、删除或更新 BDD 的 **Given / When / Then** 场景，严格遵守 `write-user-story` 的格式标准。

## When
- 对话中引入了新的边缘情况、错误修复或功能增强。
- 用户说“我们还需要添加……”、“如果……会怎样？”或“我们也应该处理……”。
- 用户明确要求“改进用户故事”或“更新故事”。
- 智能体意识到新讨论的需求使得现有用户故事变得不完整或过时。

## Where
- 修改包含用户故事的现有 markdown 文件（如 `README_UserStories.md`），或将更新后的故事输出到聊天中。

## Why
- 需求会在讨论中演变，而文档经常会变得陈旧。
- 保持 BDD 验收标准与最新对话完美对齐，可以防止测试出现遗漏。
- 主动更新可节省时间，并减轻开发人员试图追踪变更时的认知负担。

## Inputs
- **现有用户故事**（必填）：用户故事的当前版本（来自文件或聊天上下文）。
- **新改进点/功能**（必填）：对话中引入的新需求、边缘情况或工作流变更。

## Output
严格遵循 `write-user-story` 格式的更新后的用户故事：

```markdown
## User Story: <简明的标题>

**As a** <角色>,
**I want** <期望的能力>,
**So that** <业务价值>.

### Acceptance Criteria

#### Scenario 1: <场景标题>
- **Given** <前置条件>
- **When** <执行的动作>
- **Then** <预期结果>

### Notes
- <任何假设、未解决的问题或超出范围的项目，包括关于最近改进的说明>
```

## Constraints
- **保留现有场景**：不要重写未受新改进影响的场景。只修改需要改变的地方。
- **严格遵守 BDD 格式**：每个场景必须至少有一个 **Given**，一个 **When**，和一个 **Then** 步骤。使用 **And** 连接步骤。
- **杜绝捏造**：只添加最近对话支持的场景。如果改进点含糊不清，请在更新前询问澄清问题。
- **可追溯性**：清楚地表明哪些内容被添加或改进了（例如，在 Notes 部分说明，或高亮显示新场景）。

## One More Thing
如果不清楚要更新*哪个*用户故事，或者新改进点与核心需求相冲突，请在继续操作前停止并向用户寻求澄清。

## How

### Step 1: Detect Change & Identify Target (检测变更并识别目标)
1. 注意对话中何时就新功能、改进或边缘情况达成了共识。
2. 找到涵盖该领域的现有用户故事（例如：检查 `README_UserStories.md` 或近期的聊天上下文）。

### Step 2: Analyze Impact (分析影响)
1. 确定新功能是否改变了核心目标（即 "I want" / "So that"）。
2. 识别现有场景是否需要修改（例如，“快乐路径”现在有了一个新条件），或者是否必须创建全新的场景。

### Step 3: Apply `write-user-story` Guidelines (应用撰写指南)
1. 保持标准的 **As a / I want / So that** 结构。
2. 起草涵盖新功能的新的 **Given / When / Then** 场景。
3. 确保场景的原子性：每个场景只描述一个行为。

### Step 4: Deliver the Update (交付更新)
1. 输出完整的、更新后的用户故事。
2. 提供一个简短的摘要，说明为了适应对话中产生的改进，具体修改或添加了哪些场景。
3. 如果是在代码库中操作，将其保存到目标文件中。

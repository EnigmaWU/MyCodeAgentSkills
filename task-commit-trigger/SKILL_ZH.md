# 任务提交触发器 (Task Commit Trigger)

## 谁 (Who)
刚完成一个任务，希望生成高质量、结构化 Git 提交信息的开发者或智能体。

## 是什么 (What)
当任务刚完成时，自动触发或主动提醒开发者：“我们要提交刚刚完成的工作吗？”(Shall we commit just completed work?)。如果用户同意，它将生成结构化的 Git 提交信息（包含 WHAT、HOW、WHY），并在确认后执行提交。

## 什么时候 (When)
- 当对话上下文中讨论的任务或用户需求被满足/完成时。
- 用户表示“这个任务完成了”、“看起来不错”、“保存进度”，或明确要求提交代码时。
- 智能体在交互过程中完成了一个逻辑工作单元时。

## 在哪里 (Where)
适用于工作区中的本地 Git 仓库。

## 为什么 (Why)
- 优秀的提交信息对项目的可维护性至关重要。
- 在任务刚完成时立刻记录 WHAT、HOW 和 WHY，能最准确地保留上下文。
- 节省开发者的时间和认知成本。

## 输入 (Inputs)
- **代码变更**：任务期间修改的文件的差异（diff）或描述。
- **任务上下文**：刚完成的任务的目标。

## 输出 (Output)
- 结构化的提交信息，包含：
  - **WHAT**：简短总结解决了什么问题或添加了什么功能。
  - **HOW**：简要解释技术上是如何修改的。
  - **WHY**：说明采用这种方案或设计决策背后的原因。
- 执行 Git 提交（需用户批准）。

## 约束 (Constraints)
- 不要提交与刚完成的任务无关的更改。
- 提交信息应简明扼要但信息量丰富。
- 确保 WHY 部分侧重于设计决策，而不仅仅是重复 HOW 的内容。

## 还有一件事 (One More Thing)
如果不清楚、有遗漏或冲突，请停下来并询问用户再继续。

## 怎么做 (How)
1. **主动提醒**：在对话中完成任务时，自动触发或提醒开发者：“我们要提交刚刚完成的工作吗？”。
2. **分析变更**：如果用户同意，查看 Git diff 和已完成的任务上下文。
3. **草拟信息**：构建包含 WHAT、HOW 和 WHY 清晰段落或要点的提交信息。
4. **向用户展示**：将草拟的提交信息展示给用户以供最终确认。
5. **提交代码**：获得批准后，执行 Git 提交命令。
